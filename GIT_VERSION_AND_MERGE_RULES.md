# Git 提交、版本迭代与分支合并规范

## 1. 基本要求

当任务中明确要求“提交并推送到仓库”时，必须完成以下事项：

1. 检查本次修改内容。
2. 判断本次修改属于：
   - 功能新增
   - Bug 修复
   - 功能优化
   - 重构调整
   - 文档更新
3. 根据修改类型迭代项目版本号。
4. 提交 Git Commit。
5. 推送到远程仓库。
6. 清楚说明提交分支、版本号和修改内容。

---

## 2. 版本号规则

版本号统一使用：

```text
主版本号.次版本号.修订号
```

例如：

```text
1.2.3
```

### 主版本号

存在重大升级、架构调整或不兼容修改时增加。

```text
1.5.2 → 2.0.0
```

### 次版本号

新增功能，且保持原有功能兼容时增加。

```text
1.5.2 → 1.6.0
```

### 修订号

修复 Bug、优化细节或处理小问题时增加。

```text
1.5.2 → 1.5.3
```

### 多种修改同时存在

按照影响最大的修改类型决定版本号：

```text
重大升级 > 功能新增 > Bug 修复 > 文档更新
```

禁止只修改代码、不更新版本号。

---

## 3. 提交类型

Git Commit 必须标明修改类型。

| 类型 | 说明 |
| --- | --- |
| `feat` | 新增功能 |
| `fix` | 修复 Bug |
| `perf` | 性能优化 |
| `refactor` | 代码重构，不新增功能、不修复 Bug |
| `ui` | 界面或交互调整 |
| `docs` | 文档修改 |
| `style` | 代码格式调整 |
| `build` | 构建、打包、依赖调整 |
| `chore` | 其他维护性修改 |

提交格式：

```text
类型: 简要说明
```

示例：

```text
feat: 新增小米 MiMo API 用量查询
fix: 修复 DeepSeek 今日金额不显示的问题
ui: 优化贴边自动隐藏动画
build: 减小 Windows 安装包体积
docs: 更新 v1.2.0 使用说明
```

禁止使用以下模糊提交信息：

```text
修改一下
更新代码
修复问题
提交
优化
```

---

## 4. 版本更新说明

每次版本迭代必须写清楚本次属于哪种更新。

示例：

```markdown
## v1.2.0

更新类型：功能更新

### 新增

- 新增小米 MiMo API 用量查询。
- 新增近 7 天消费趋势展示。

### 修复

- 修复 DeepSeek 部分返回数据解析异常。
- 修复今日消费金额不显示的问题。

### 优化

- 优化悬浮球贴边自动隐藏动画。
- 优化程序打包体积。
```

只修复 Bug 时：

```markdown
## v1.2.1

更新类型：Bug 修复

### 修复

- 修复今日金额统计错误。
- 修复窗口贴边后无法恢复的问题。
```

## 5. 分支命名

分支名称必须能够看出用途。

```text
feature/功能名称
fix/问题名称
refactor/模块名称
release/版本号
hotfix/紧急问题名称
```

示例：

```text
feature/mimo-api
fix/deepseek-daily-cost
refactor/usage-service
release/1.2.0
hotfix/startup-crash
```

禁止使用：

```text
test
new
temp
dev2
修改
```

## 6. 分支合并请求规范

创建合并请求时，标题和说明必须清楚。

### 合并请求标题

```text
类型: 简要说明
```

示例：

```text
feat: 新增 MiMo API 用量查询
fix: 修复 DeepSeek 金额统计异常
release: 发布 v1.2.0
```

### 合并请求说明

```markdown
## 修改类型

- [x] 功能新增
- [ ] Bug 修复
- [ ] 功能优化
- [ ] 代码重构
- [ ] 文档更新

## 修改内容

- 新增 MiMo API 查询适配。
- 增加 MiMo 平台用量展示。
- 更新相关配置说明。

## 影响范围

- API 数据查询模块。
- 用量统计面板。
- 配置文件。

## 验证结果

- 已完成本地运行验证。
- 已完成主要功能测试。
- 未发现阻塞性问题。

## 版本变化

```text
v1.1.0 → v1.2.0
```
```

禁止只填写：

```text
合并代码
修改完成
请审核
更新
```

---

## 7. 合并前检查

提交或创建合并请求前必须确认：

- [ ] 本次修改能够正常运行。
- [ ] 没有明显报错。
- [ ] 没有提交临时文件、日志、缓存或敏感信息。
- [ ] 已更新版本号。
- [ ] 已写明属于功能新增、Bug 修复或其他类型。
- [ ] Commit 信息清楚。
- [ ] 分支名称清楚。
- [ ] 合并请求标题和说明清楚。
- [ ] README、CHANGELOG 或版本说明已按需更新。
- [ ] 已确认推送到正确的远程分支。

---

## 8. 推送完成后的汇报格式

完成提交和推送后，必须明确汇报：

```markdown
已完成提交并推送。

- 当前版本：v1.2.0
- 更新类型：功能更新
- 提交分支：feature/mimo-api
- 目标分支：main
- Commit：feat: 新增 MiMo API 用量查询
- 推送状态：成功

主要修改：

- 新增 MiMo API 查询支持。
- 修复 DeepSeek 今日金额不显示。
- 优化贴边自动隐藏动画。
- 更新 README 和版本说明。
```

不得只回复：

```text
已完成
已推送
修改好了
```

---

## 9. 执行原则

当用户要求提交、推送、合并或发布版本时：

1. 不得跳过版本号更新。
2. 不得使用模糊的 Commit 信息。
3. 不得使用含义不清楚的分支名称。
4. 必须写清楚是功能新增、Bug 修复、优化还是重构。
5. 必须说明源分支、目标分支和合并内容。
6. 必须在推送后报告实际执行结果。
7. 如果没有仓库权限、远程地址异常或推送失败，必须明确说明原因，不得假装成功。
